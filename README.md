## 🚀 接口自动化测试执行与 Allure 可视化报告交付

本框架采用 Pytest 作为核心测试驱动引擎，联动 Allure 实现了生产级的质量看板交付。为了打造极致干净的开源代码仓，全量测试中间产物（`allure-results/`）已在 `.gitignore` 中执行了硬隔离拦截。

### 1. 环境依赖就绪 (Prerequisites)
进入本项目的接口自动化核心目录激活你的 Python 虚拟环境，并一键补齐企业级跑测依赖：
```bash
cd 1-API-Automation
pip install -r ../requirements.txt
```

### 2. 一键触发公网实战跑测 (Execute Automated Tests)
执行以下标准命令，框架将瞬间穿透至互联网，对**若依官方线上演示系统**发起完整的全链路接口跑测。框架在执行过程中会自动完成动态验证码解耦与分布式 Token 拦截，并将结构化测试数据沉淀至 `allure-results` 文件夹中：
```bash
# 执行全部接口自动化用例（包含登录认证与用户信息级联获取）
python -m pytest -vs tests/ --alluredir=allure-results
```

### 3. 本地一键拉起 HTML 质量看板 (Generate & Serve Allure Report)
由于跑测生成的原始数据均为非直视的 JSON 级联分片，请确保你的本地系统（Windows/Mac）已配置好 Allure 命令行环境，随后在 `1-API-Automation` 目录下执行以下命令。系统将自动在本地沙盒中编译并拉起一个惊艳的、可交互的可视化 HTML 报告网页：
```bash
allure serve allure-results
```
*(💡 提示：Allure 看板包含用例成功率饼图、严密的 @allure.step 业务执行序列、响应体 JSON 动态审计以及失败时的全量屏幕快照。)*

#### 🛡️ 框架核心高光技术栈实现说明：
1. **API Object Model (接口对象模型) 分层架构**：项目摒弃了流水账式的线性脚本编写，采用工业级 POM 思想进行服务解耦。将底层接口协议定义与网络行为内聚在 `api_objects/` 专属对象层中，测试用例层（`tests/`）仅关注测试数据驱动与核心资产断言，具备极高的框架可读性与后期复用扩展性。
2. **Pytest Fixture 会话级数据共享与鉴权隔离**：在 `conftest.py` 中全手工打造了 `scope="session"` 的全局登录生命周期夹具。完美实现了**全套件仅需登录认证一次，后续多业务模块（如 `test_user.py` 获取用户信息）自动免密共享身份 Token 鉴权**的高级用例组织设计，彻底根治了自动化测试频繁登录导致的性能损耗与账号防刷拦截问题。
3. **高级单元 Mock 测试技术与工程级防御机制**：针对若依官方演示环境近年来高频安全策略升级、彻底关闭万能验证码后门导致接口阻断的痛点，框架深度融入 `pytest-mock` 组件。在 `session_mocker` 与用例级 `mocker` 双重作用域下，实现全链路网络挡板的优雅拦截与标准 JSON 数据伪造。**此项设计有力向面试官证明了在第三方接口不可控、外部依赖不稳定（或处于并行开发期）的复杂企业环境下，如何通过 Mock 机制确保 CI/CD 持续集成链路 100% 稳定通畅的架构解耦思维。**
4. **多层级业务故事线装饰器与动态审计**：测试用例层深度融入了 `@allure.epic`（若依系统）、`@allure.feature`（认证/用户模块）、`@allure.story`（特定场景）等大厂级纵深原语。配合代码内 `allure.attach` 钩子，将动态请求的 Payload 载荷、响应体 JSON 以及 UUID 凭证实时抽稀并挂载至测试步骤树中，实现了非结构化测试成果向可读性极高、带步骤展开树（Steps）的工业级质量看板转化。



## 🚀 性能测试执行与 16GB 本地单机容量演练

本框架在 `3-Performance-Testing` 目录中开辟了性能压测中心。为了在不依赖复杂压测机集群的前提下，在本地（16GB 内存/CPU）跑出高质量、具备证明闭环的压测作品，项目采用了 **Locust (基于 Gevent 协程的高并发压测引擎)**。

框架实现了“**Pytest 驱动 + Locust 施压 + 施压机硬件自监控 + 报告无缝塞入 Allure/HTML**”的闭环链路，兼顾了压测效率与报告的一体化交付。

### 1. 环境依赖就绪 (Prerequisites)
切换至性能测试专属核心目录，激活你的 Python 虚拟环境，并一键补齐性能压测与硬件监控专属依赖：
```bash
cd 3-Performance-Testing
pip install pandas psutil locust
```

### 2. 双通道执行模式：自由选择你的作品秀

#### 🛠️ 通道 A：一键触发全自动压测与施压机监控 (最佳亮点推荐)
运行该脚本，系统将以 `Headless (无界面)` 模式快速撬动 **100 并发用户**，穿透压测至公网性能靶场。与此同时，脚本会异步拉起线程，每秒动态审计你 16GB 施压机自身的 CPU 与内存，防止“假瓶颈”发生。压测结束后，会自动在目录下生成极为专业的本地 HTML 看板以及监控报表：
```bash
python run_perf_test.py
```
* 查看成果：跑完后，直接双击打开目录下生成的 `perf_report.html`，即可在浏览器中查看全量 RPS 动态曲线、吞吐量表格和百分位数响应时间。

#### 📊 通道 B：全链路看板整合，将性能数据并入 Allure 看板
如果你想将接口自动化和性能测试统一在一个大盘中展示，框架编写了 `test_perf_bridge.py` 桥接用例。它会在跑完 Locust 后，自动将 CSV 指标读取并实时渲染成 HTML 节点塞进 Allure 树中：
```bash
# 触发 Pytest 性能用例收集
pytest test_perf_bridge.py --alluredir=allure-results

# 编译并拉起包含“性能测试中心”的 Allure 综合看板
allure serve allure-results
```

---

### 🛡️ 性能压测核心高光技术栈实现说明：

1. **Gevent 协程高并发选型与单机 16GB 环境适配**：
   相较于 JMeter 沉重的多线程机制（极其消耗施压机内存），本框架选择基于 Python 协程的 Locust 引擎。在 `locustfile.py` 中，通过严密的 `between(1, 2)` 思考时间控制与多接口 `@task(weight)` 权重配比（3:1 模拟真实高频查询与慢接口混合场景），成功在 16GB 单机环境下平稳模拟高负载，展现了用极低硬件成本换取高并发吞吐量的架构思维。
2. **异步 `psutil` 施压机资源防御性审计（核心高光点）**：
   大部分初学者做性能压测只看目标服务器状态，往往忽略了“施压机自身 CPU/内存先耗尽导致测试数据不准”的痛点。我们在 `run_perf_test.py` 中全手工编写了多线程硬件监控，压测时每秒记录施压机指标并输出为 `perf_report_system_metrics.csv`。向面试官有力证明了**你具备‘排除因施压机自身瓶颈导致吞吐量上不去’的工业级闭环排查思维**。
3. **Windows 异步事件循环（Proactor/Selector）兼容性解耦**：
   针对 Windows 环境下使用 Python 3.12 运行 Gevent 容易触发的底层绿线（Greenlet）清理异常与时序断层问题，框架在 `test_perf_bridge.py` 顶部强行注入了 `WindowsSelectorEventLoopPolicy` 事件循环策略补丁，并在子进程执行完毕后设置了 `time.sleep(3)` 的稳健刷盘缓冲。**此项底层兼容性修复，充分展现了你具备独立排查并解决多平台环境冲突的底层代码自理能力。**

## 📊 项目实战篇：Locust 桌面端实操与 5542 次高并发深度性能分析

在性能压测闭环交付中，框架除了支持一键命令行（Headless）跑测，亦完美适配 **Locust 图形桌面化（Web UI）交互大盘**。以下为模拟 100 并发虚拟用户、全链路发起 **5542 次高并发冲击**后的真实看板数据审计与根因分析（RCA）报告[GET]。

### 1. 桌面端仪表盘拉起与实操演练 (Web UI Operation)
在 `3-Performance-Testing` 目录下执行去参数化的纯净启动命令，框架会自动拉起本地轻量化 Web 服务器：
```bash
locust -f locustfile.py
```
* **访问入口**：保持终端运行，在任意浏览器访问 `http://localhost:8089`[GET]。
* **参数配置**：最大并发用户数（Users）配置 `100`，激增速率（Spawn Rate）配置 `10`（即 10 秒内匀速加满 100 人），目标 Host 指向性能靶场 `https://httpbin.org`[GET]。
* **图表特性说明**：离线导出的静态 HTML 报告与 8089 实时运行的动态 ECharts 大盘在曲线上有样式差异（离线报告为了保证零网格离线浏览，进行了数据点静态降采样与重排版），但**核心峰值、延迟拐点与吞吐数据 100% 绝对一致**[GET]。

### 2. 5542 次压测大盘深度性能审计报告 (Data Analysis)

#### ⚙️ 维度一：流量模型合规性审计 (Traffic Model Validation)
* **数据断言**：全链路累计触发 **5542 次**有效请求，持续高频轰炸 **2 分钟 12 秒**[GET]。其中，核心查询接口共执行 **4166 次**（31.55 RPS），慢 SQL 模拟接口共执行 **1376 次**（10.42 RPS）[GET]。
* **架构结论**：两接口的流量占比为精准的 **75.1% vs 24.9%**，完美印证了代码中通过 `@task(3)` 与 `@task(1)` 设计的 **3:1 核心/次要混合业务负载模型**[GET]。

#### 🚨 维度二：核心瓶颈诊断与 502 Bad Gateway 根因分析 (RCA)
* **异常现象**：在高负载稳定期，系统触发了 **40 次应用层报错**（核心接口 24 次，慢接口 16 次），失败率约 0.7%，报错明细指向：`HTTPError('502 Server Error: Bad Gateway...')`[GET]。
* **多图表联动诊断（核心高光分析点）**：
  1. 联动分析 **Charts 三曲线走势**，在用户数加满至 100 并持续施压的某时间点（约 7:49:45 PM），第一张图的 **RPS 吞吐量发生了明显的向下凹陷（瞬间塌陷跌落）**[GET]；
  2. 相同时间点，第二张图的 **Response Times 响应时间曲线发生剧烈向上抬头（延迟突增）**[GET]。
* **技术根因剖析**：502 报错意味着前端反向代理（如 Nginx）在转发请求时，后端的 Web 服务器由于瞬间高并发冲击，导致 **Tomcat/Node 线程池瞬间耗尽，或后端的慢接口响应超时，进而诱发了网关崩溃**[GET]。这证明 **100 并发负载已逼近该性能靶场的单机最大容量拐点**[GET]。

#### ⏱️ 维度三：长尾延迟劣化与均值掩盖效应分析 (Long-tail Effect)
* **长尾指标变动**：
  * **核心查询接口**：虽然平均响应时间仅为 447ms，但 **P99 百分位数（长尾延迟）暴增至 2400ms（2.4秒）**，最大延迟极限达到了 21.8 秒[GET]！
  * **慢 SQL 模拟接口**：平均响应时间 1.5 秒，但 **Max 最大处理耗时飙升至 26.6 秒**[GET]。
* **调优策略建议**：系统存在严重的“平均值掩盖极值”的长尾效应。有 1% 的高端用户体验发生了指数级劣化。在工业级项目中，应当开启数据库慢日志监控（Slow Log Query），排查是否存在全表扫描导致的数据库排他锁锁等待，或者高并发下 JVM 频繁触发 **Full GC (Stop-The-World)** 导致的系统短暂假死[GET]。
