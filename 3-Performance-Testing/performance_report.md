# 📈 若依/本地服务基准性能测试与容量演练报告

## 1. 测试背景与场景设计
本报告旨在对系统核心链路（登录、音乐列表查询、用户信息获取）进行单机容量演练，评估在本地单机环境下的性能表现。

* **施压机配置**：本地本地环境 (16GB RAM / CPU)
* **压测引擎**：Locust (Headless 协程模式)
* **并发策略**：最大并发用户数 100 User，激增速率 (Spawn Rate) 10 User/s，持续压测运行。
* **目标主机**：`http://127.0.0.1:8080` (本地后端服务)

## 3. 🚨 性能瓶颈分析与问题诊断 (核心高光区)

本次压测全链路触发了 **100% 阻断性失败**，全链路 RPS 停留在 14.96，响应时间突增至 2 秒以上。通过比对 `Failures Statistics` 与控制台日志，定位出以下两大底层瓶颈：

### 瓶颈一：目标服务器端口未监听 (ConnectionRefusedError)
* **现象**：`/api/v1/login` 与 `/music/list` 报错明细均为：`ConnectionRefusedError(10061) No connection could be made because the target machine actively refused it.`
* **诊断**：该错误属于典型的**网络传输层阻断**。由于本地单机环境下未拉起对应的后端服务进程，导致施压机发出的 TCP 三次握手请求在指定端口（8080）被操作系统直接回绝（RST）。
* **衍生影响**：Locust 内部由于连接持续重试与超时等待，导致虚拟用户产生严重的**线程/协程阻塞**，使得平均响应时间被动拉长至 2300ms 以上（此数据并非真实的服务器处理耗时，而是客户端网络等待耗时）。

### 瓶颈二：应用层状态码异常与断言拦截 (Status Code: 0)
* **现象**：`/user/profile` 接口抛出 `CatchResponseError: Got unexpected status code: 0`。
* **诊断**：状态码 `0` 在 HTTP 协议中并非标准服务端返回码，通常代表**请求未成功发出或在客户端被提前终止/拒绝连接**。由于底层连接失败，Locust 的 `client.get` 无法获取有效的 Response 对象，触发了脚本中严密的 `if response.status_code != 200` 工业级防御断言，成功拦截了脏数据。

## 4. 🛠️ 性能调优建议与闭环验证策略

针对上述暴露的单机瓶颈，在后续的持续集成与容量交付中，规划采取以下调优手段：

1. **服务生存状态健康检查 (Health Check) 拦截机制**：
   在自动化测试或 CI/CD 流水线驱动压测前，增加前置探针脚本。优先通过 `curl -I` 或 python `requests` 对目标端口进行网络拨测。若返回非 200，则直接熔断压测链路，防止空跑压测浪费施压机资源。
2. **连接池与超时机制优化**：
   在 `locustfile.py` 中，针对网络不可达风险，应当显式配置客户端超时时间（例如 `timeout=5.0`），防止异常连接无限期挂起协程，确保压测引擎自身的稳健性。
3. **环境解耦闭环方案**：
   * **方案 A（本地服务起服）**：确保在压测前，本地 Docker 容器或 SpringBoot 后端进程已正常 Standby，且数据库连接池（如 Druid）配置充足。
   * **方案 B（外部靶场穿透）**：若本地无服务，将 `--host` 切换至公网标准性能测试靶场（如 `https://httpbin.org`），验证全链路各方法权重（75% vs 25%）与流量模型的正确性。
